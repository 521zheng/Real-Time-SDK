Transport API Consumer Training Example
 
Example
Description
Consumer_Module_1a_Connect
The first step of a consumer application is to establish a network connection with its peer component (i.e., another application with which to interact). An OMM consumer typically creates an outbound connection to the well-known hostname and port of a server (Interactive Provider or ADS). The consumer uses the Transport.connect() method to initiate the connection and then uses the Channel.init() method to complete channel initialization.
Consumer_Module_1b_Ping
Ping or heartbeat messages indicate the continued presence of an application. Once the consumer’s connection is active, ping messages must be exchanged with a negotiated timeout (retrieved using the Channel.info() method). The connection will be terminated if a ping heartbeat are not sent or received within the expected time frame.
Consumer_Module_1c_ReadWrite
After the channel initializes, the state of the channel (ChannelState) is ACTIVE and an application can now send or receive data.
Consumer_Module_2_Login
Applications authenticate using the Login domain model. An OMM consumer must authenticate with a provider using a login request prior to issuing any other requests or opening any other streams. After receiving a login request, an interactive provider determines whether a user is permissioned to access the system. The interactive provider sends back a login response, indicating whether the consumer is granted access.
Consumer_Module_3_Directory
The Source Directory domain model conveys information about all available services in the system. An OMM consumer typically requests a source directory to retrieve information about available services and their capabilities. This includes information about supported domain types, the service’s state, the quality of service (QoS), and any associated item group information.
Consumer_Module_4_Dictionary
Consumer applications often require a dictionary to encode or decode data. This dictionary typically defines type and formatting, and directs the application in encoding or decoding data. Content that uses the FieldList type requires the use of a field dictionary (usually RDMFieldDictionary, though it can be a user-defined or modified-field dictionary).
A consumer application can choose whether to load necessary dictionary information from a local file or download the information from the provider (if available).
Consumer_Module_5_ItemRequest
After a consumer logs in and obtains source directory and dictionary information, it can request additional content. When issuing the request, the consumer specifies the serviceId of the desired service along with a streamId. Requests can be sent for any domain using the formats defined in that domain model specification. This example illustrates how to make a Market Price level I data Item request to obtain data from a provider.
